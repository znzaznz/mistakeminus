# 计划 C · 周二复识图片识题（VL/OCR）

> 优先级：低（有 API 成本、产出不确定，建议最后做）
> 来源：`media/周周老师-2026强化班—…周二复识{129667,633088,673715,734192}.pdf`（4 份，**图片型**）
> - 129667 = 第五章合同 · 633088 / 673715 = 第一部分 · 734192 = 第二章

## 1. 内容形态

**图片型 PDF**——几乎没有文字层（实测每页文字 ~32 字，全是页眉页脚），正文是图片（扫描/截图）。共约 14+6+10+21=51 页、300+ 张图。

直接用 fitz 抽文字拿不到内容，**必须走视觉识别（VL/OCR）**。

## 2. 价值与取舍

「复识」是复习串讲材料，价值不如集训题库直接。考虑到：

- 需调 DashScope VL 逐页识图 → **有 API 成本**
- 识别结果质量不确定，可能需要人工校对
- 内容可能与讲义/集训重叠（复习性质）

**建议**：先不急着全量识别。等 A、B 落地后评估是否还缺这部分内容，再决定是否投入。

## 3. ⚠️ 铁律：VLM 只发一次，一次榨干所有字段

**整个项目这批素材里，唯一花 VLM token 的就是这个计划**（A、B 是文字型，0 VLM）。因此每张图**这辈子只发一次 VLM**，且这一次就把后续**所有功能**会用到的字段**全部一次性提取**，绝不为了不同用途（先提题目、再提答案、再分类知识点……）反复发同一张图。

### 单次提取必须覆盖的全字段（一张图一次出齐）

按下游功能倒推，一次 VLM 调用要求模型对每页/每题同时返回：

| 字段 | 下游用途 |
|---|---|
| `page_raw_text` 整页 OCR 全文 | **兜底**：即使将来需要现在没结构化的东西，也有全文可用，**永不回头重识** |
| `material_type`（题目 / 讲解 / 混合） | 分流入题库 or 知识层 |
| `chapter` 章节 | 归类、按章刷题 |
| `kaodian` 考点 / `knowledge_point` 知识点候选 | S6 归类、S7 薄弱点 |
| `question_type` 题型（单选/多选/判断/不定项） | 刷题判分 |
| `stem` 题干 / `options` 选项 A–D | 刷题主体 |
| `correct_answer` 正确答案 | 判分（图里有就提，没有标缺） |
| `explanation` 解析 | 错题讲解、S8 相似题素材 |
| `difficulty` 难度（可推断则给） | S9 排程权重 |
| `lecture_text` 讲解正文（非题目页） | 知识层、S8 素材 |
| `confidence` 模型自评置信 | 人工确认排序 |

**一个结构化 JSON schema 把上面全包进去，一次 prompt 全要。** 这样无论以后接 S6/S7/S8/S9 哪个功能，都直接读这份已存结果，不再碰 VLM。

### 流程

1. fitz 把每页渲染成图（`page.get_pixmap`，`pdftoppm` 不可用，用 PyMuPDF 自带渲染）。可在 token 与精度间权衡分辨率。
2. **一次性批量**把 4 份全部 51 页按上面的全字段 schema 喂 VL，**结果原样落盘**（如 `data/vlm_fushi_raw.jsonl`，每页一条），作为**唯一真源**。
3. 后续所有入库 / 分类 / 生成，都从这份落盘 JSON 读，**不再调 VLM**。
4. 按 `material_type` 分流：题目→题库、讲解→知识层。
5. **全部标低置信、走人工确认**，不静默入库（图片识别易错，绝不重演 S5 的高置信静默写入 + 删原始）。

## 4. 验证（图片识别尤其要严）

- [ ] **先 1 份试识**核对全字段抽取质量 + 看单页 token 消耗，达标 + 估清全量成本后，**一次性把剩余全部识别完**（不要分多轮试探，那才是浪费 token）
- [ ] VLM 原始输出**完整落盘**（`data/vlm_fushi_raw.jsonl`），确认字段齐全可支撑 A~C 全部下游，再做入库
- [ ] 识别产物全部待确认状态；原图保留（`media/` 不删，落盘 JSON 也在，**永不需重识**）

## 5. 风险

- VL 识别错字/漏题 → 必须人工抽检，不可信任直接入库。
- 与已有内容重复（复识本是复习材料）→ 入库前按题干/知识点去重。
- **字段提漏导致回头重识** → 上线前用「全字段 schema」对 1 份做完整核对，确认无遗漏字段再批量，避免二次 VLM。

## 6. 工作量

约 1 个 issue（依赖 S10 管线）：先 1 份试点定 schema 与成本 → 一次性批量识别全部并落盘 → 从落盘结果入库。优先级排在 A、B 之后。
