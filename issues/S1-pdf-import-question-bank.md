# S1 — PDF 导入成结构化题库

- **类型**：AFK
- **批次**：第一批（最小闭环骨架）
- **状态**：✅ 已完成。管线 + 测试 + 真 PDF 跑通（123 道真题入库，数据干净）。详见底部验收记录。
- **覆盖用户故事**：1–6（题库与资料导入）

## What to build

一个**离线运行的 Python 提取脚本**：输入讲义 PDF，输出结构化题库写入 SQLite。

端到端流程：

1. 用 PyMuPDF 把每页**整页渲染成图**（`page.get_pixmap`）。
2. 整页图喂给视觉大模型（`QWEN_VL_MODEL`，默认 `qwen-vl-max`），返回结构化 JSON：题干 / 选项 / 正确答案 / 解析 / 所属考点 / 年份 / 题型（单选/多选/判断）/ 是否含图 / 置信度。
3. 对带图的题，用嵌入图**坐标框**（`page.get_images` + `page.get_image_rects`）定位，与题干文字 y 坐标比对确定归属，再用**裁剪渲染**（`page.get_pixmap(clip=bbox)`）重渲染出该图，存入该题「配图」字段（本地文件夹存图，DB 存相对路径）。
4. 结果写入 SQLite `questions` 表；低置信度的题打上「需人工确认」标记。

**关键约束（来自 `_demo/` 实测验证，不要踩坑）**：
- 配图一律用「按坐标裁剪渲染」获取。
- **禁止**用 `Pixmap(doc, xref)` 直接抠原始嵌入图对象 —— 实测会因透明遮罩/颜色空间得到全黑图。

VLM 调用在单测里 **mock 掉**（不打真实 API）。真实 key 到位后，对项目里那两个真 PDF 跑一次作为人工验收。

## Acceptance criteria

- [ ] `questions` 表 schema 落地，至少含：题目ID、题目文本、选项、所属章节、所属考点、题型、难度、年份、正确答案、解析、配图（路径列表，可空）、来源（标记为 PDF导入）、置信度/需确认标记
- [ ] 整页渲染产物存在且尺寸正确（可在 mock 下测）
- [ ] 带图题能取到嵌入图坐标框；裁剪渲染产物非空且**非全黑**（可在 mock 下测）
- [ ] VLM 返回的 JSON 通过 schema 校验；不合格的进「需人工确认」队列
- [ ] 单测全程 mock VLM，不依赖真实 API key
- [ ] 脚本跑完打印「成功识别 N 道题、M 道需人工确认」
- [ ] （人工验收，需真实 key）对两个真 PDF 跑一遍，约 60 道题正确入库，带图题配图正确显示
